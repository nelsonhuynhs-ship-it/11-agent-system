Workflow:
1. Run `git diff --staged --stat` to see file summary
2. Run `git diff --staged` to see actual changes
3. Analyze: nature of change (feat/fix/refactor/docs/test/chore)
4. Output message in this exact format:

```
<type>(<scope>): <subject under 72 chars>

<body explaining WHY, not WHAT — wrap at 72 chars>

<optional footer: BREAKING CHANGE / Closes #N / Refs #N>
```

Type rules:
- `feat`: new functionality
- `fix`: bug fix
- `refactor`: code change without behavior change
- `docs`: documentation only
- `test`: test additions/changes
- `chore`: tooling, deps, config
- `perf`: performance improvement
- `style`: formatting only

Constraints:
- Subject in lowercase except proper nouns
- No period at end of subject
- Body optional but recommended for non-trivial changes
- Output ONLY the commit message text, nothing else
- Don't include "Generated by Claude" or similar
